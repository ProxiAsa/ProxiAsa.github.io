Ferme RDS Haute Disponibilité
2 Brokers — 4 Session Hosts — UPD

Déploiement d'une ferme Remote Desktop Services haute disponibilité pour un client entreprise d'Aintranet. Architecture redondée avec deux RD Connection Brokers en actif/actif et quatre Session Hosts répartis en deux collections, gestion des profils utilisateurs via User Profile Disks (UPD).

Réalisé
📅 2025
🏢 Aintranet — Client entreprise
RDS Windows Server 2022 Active Directory UPD Connection Broker HA RemoteApp SQL Server

Contexte & objectifs

Dans le cadre d'une mission pour Aintranet, j'ai été chargé de déployer une ferme RDS pour un client entreprise nécessitant une haute disponibilité. Contrairement à une installation mono-broker, cette architecture utilise deux RD Connection Brokers synchronisés via une base SQL partagée, garantissant la continuité de service en cas de panne d'un broker. Les profils utilisateurs sont gérés par User Profile Disks (UPD), permettant à chaque utilisateur de retrouver son environnement identique quel que soit le Session Host sur lequel il est routé.

Contexte

Client entreprise — besoin de HA et de performances réseau optimisées

Session Hosts

4 serveurs répartis en 2 collections de 2 RDSH

Brokers

2 RD Connection Brokers en actif/actif avec BDD SQL partagée

Profils utilisateurs

User Profile Disks (UPD) — disques VHD montés à la connexion sur chaque Session Host

Architecture déployée

A Coll. B — Applications publiées par collection

📁 User Profile Disks (UPD) : Les UPD sont des disques virtuels VHD stockés sur un partage réseau et montés automatiquement à l'ouverture de session sur le Session Host. Chaque utilisateur possède son propre fichier .vhdx contenant son profil complet. Contrairement aux profils itinérants classiques, le profil n'est pas copié — il est monté directement, ce qui rend les connexions plus rapides et évite les conflits de synchronisation.

Étapes de déploiement

  • 01
    Préparation AD & DNS — Création des OUs dédiées (Serveurs RDS, Utilisateurs RDS), enregistrement DNS des brokers, attribution des licences CAL utilisateur dans AD.
  • 02
    Installation SQL Server — Déploiement de SQL Server (Express ou Standard) sur un serveur dédié. Création de la base de données partagée pour la haute disponibilité des brokers. Ajout des comptes machines Broker-01 et Broker-02 avec droits db_owner.
  • 03
    Déploiement des rôles RDS — Installation via Server Manager des rôles : RD Connection Broker (x2), RD Session Host (x4), RD Gateway, RD Web Access sur les serveurs désignés.
  • 04
    Configuration HA des brokers — Dans le Gestionnaire de serveur > Services Bureau à distance, configurer la haute disponibilité du broker : renseigner la chaîne de connexion SQL, le DNS RoundRobin partagé entre Broker-01 et Broker-02.
  • 05
    Création des collections — Création de deux collections de sessions : Collection A (SH1 + SH2) pour les applications métier, Collection B (SH3 + SH4) pour les usages bureautiques. Configuration du load balancing par poids entre les Session Hosts.
  • 06
    Configuration des User Profile Disks (UPD) — Activation des UPD dans les propriétés de chaque collection RDS. Création du partage réseau dédié au stockage des fichiers VHD(X) de profils. Configuration de la taille maximale des disques et des permissions NTFS sur le partage (comptes machines des Session Hosts en contrôle total).
  • 07
    Configuration RD Gateway & Web Access — Déploiement du certificat SSL, création des politiques CAP/RAP sur le Gateway, publication des RemoteApp sur le portail Web Access. Test d'accès depuis l'extérieur.
  • 08
    GPO & sécurisation — Déploiement des GPO : timeout de session, redirection imprimante, profils itinérants, restriction des droits locaux sur les Session Hosts, désactivation du bureau local.
  • 09
    Tests de basculement & profils — Simulation de panne du Broker-01 : vérification que les nouvelles sessions sont prises en charge par Broker-02. Test de reconnexion des sessions existantes. Vérification que le profil UPD est bien monté depuis le partage réseau sur les deux Session Hosts.

La ferme est opérationnelle avec une tolérance de panne sur le broker. Les User Profile Disks permettent à chaque utilisateur de retrouver son environnement identique sur n'importe lequel des quatre Session Hosts. Les deux collections assurent une séparation des usages et une meilleure gestion de la charge.
