01
Installation sécurisée de MariaDB

Sur le serveur BDD1, mettre à jour le système puis installer et sécuriser MariaDB :

apt update -y && apt upgrade -y apt install mariadb-server -y mariadb-secure-installation

L'assistant mariadb-secure-installation demande de définir le mot de passe root, supprimer les utilisateurs anonymes, désactiver la connexion root à distance et supprimer la base de test.

02
Création de l'utilisateur admin SQL

Se connecter à MariaDB et créer un utilisateur administrateur avec tous les privilèges :

mysql --user=root mysql CREATE USER 'sql_adm'@'%' IDENTIFIED BY 'Azerty_123'; GRANT ALL PRIVILEGES ON *.* TO 'sql_adm'@'%' WITH GRANT OPTION; FLUSH PRIVILEGES;

L'utilisateur sql_adm pourra se connecter depuis n'importe quelle adresse IP (wildcard %).

03
Autoriser les connexions externes

Par défaut MariaDB n'écoute que sur localhost. Modifier le fichier de configuration pour écouter sur toutes les interfaces :

nano /etc/mysql/mariadb.conf.d/50-server.cnf # Trouver la ligne bind-address et remplacer par : bind-address = 0.0.0.0
systemctl restart mariadb

Pour tester la connexion depuis un serveur distant, installer le client MariaDB et tester :

apt install mariadb-client mysql -u sql_adm -p -h [IP_BDD] -P 3306
04
Installation de Nginx sur les serveurs WordPress

Sur WP1 (et WP2), mettre à jour et installer Nginx et PHP :

apt update -y && apt upgrade -y apt install nginx -y # Vérification avec lynx apt install lynx lynx 192.168.160.21

Installer ensuite PHP et toutes les extensions nécessaires à WordPress :

apt install php php-curl php-fpm php-bcmath php-gd php-soap \ php-zip php-mbstring php-mysqlnd php-xml php-intl
05
Création de la base de données WordPress

Sur BDD1, créer la base dédiée à WordPress et l'utilisateur associé :

mysql -u sql_adm -p CREATE DATABASE NLU_Wordpress; GRANT ALL PRIVILEGES ON NLU_Wordpress.* TO "nluwordpressuser"@"%" IDENTIFIED BY "Azerty_123"; FLUSH PRIVILEGES; # Vérification SHOW DATABASES;
06
Téléchargement et configuration de WordPress

Sur WP1, télécharger et décompresser WordPress dans le répertoire web :

wget https://fr.wordpress.org/latest-fr_FR.zip apt install unzip unzip latest-fr_FR.zip -d /var/www/html

Configurer WordPress en copiant et éditant le fichier de configuration :

cd /var/www/html/wordpress cp wp-config-sample.php wp-config.php nano wp-config.php # Modifier les valeurs suivantes : define('DB_NAME', 'NLU_Wordpress'); define('DB_USER', 'nluwordpressuser'); define('DB_PASSWORD', 'Azerty_123'); define('DB_HOST', '192.168.160.31');

Attribuer les droits à l'utilisateur Nginx :

chown -R www-data:www-data /var/www/html/wordpress

Configurer le virtual host Nginx pour WordPress :

nano /etc/nginx/sites-enabled/default # Bloc server avec root /var/www/html/wordpress, # index index.php, try_files et fastcgi_pass vers php-fpm
07
Redirection via le FRONT + SSL auto-signé

Sur le FRONT, installer Nginx et créer le fichier de configuration reverse proxy :

apt install nginx -y nano /etc/nginx/sites-available/wordpress # Contenu du fichier : upstream wordpress { server 192.168.160.21; } server { listen 80; location / { proxy_pass http://wordpress; } }
ln -s /etc/nginx/sites-available/wordpress \ /etc/nginx/sites-enabled/wordpress systemctl restart nginx

Générer un certificat auto-signé (2048 bits, valable 3000 jours) :

mkdir /etc/nginx/ssl openssl req -x509 -nodes -days 3000 -newkey rsa:2048 \ -keyout /etc/nginx/ssl/front.tp.labo.key \ -out /etc/nginx/ssl/front.tp.labo.crt

Ajouter le bloc SSL dans le fichier de configuration wordpress du FRONT (port 443) et rediriger HTTP vers HTTPS. Pour corriger le problème de CSS lié au proxy, ajouter dans wp-config.php :

nano /var/www/html/wordpress/wp-config.php # Avant la ligne require_once ABSPATH . 'wp-settings.php' : if ($_SERVER['HTTP_X_FORWARDED_PROTO'] == 'https') $_SERVER['HTTPS'] = 'on';
08
Certificat Let's Encrypt

Sur le FRONT, installer Certbot et générer un certificat pour le domaine etu11.sio-carriat.com via challenge DNS :

apt install python3-certbot-nginx certbot -d etu11.sio-carriat.com --manual \ --preferred-challenges dns certonly

Suivre les instructions : ajouter l'entrée DNS TXT indiquée sur https://dns.sio-carriat.com, puis valider. Le certificat est généré dans /etc/letsencrypt/live/etu11.sio-carriat.com/. Mettre à jour la configuration Nginx pour utiliser les nouveaux certificats :

nano /etc/nginx/sites-enabled/wordpress # Modifier les lignes ssl_certificate : ssl_certificate /etc/letsencrypt/live/etu11.sio-carriat.com/cert.pem; ssl_certificate_key /etc/letsencrypt/live/etu11.sio-carriat.com/privkey.pem;

🌐 Accès final : Le site WordPress est accessible en HTTPS sur https://etu11.sio-carriat.com avec un certificat valide Let's Encrypt. Ajouter l'entrée dans le fichier hosts Windows pour les tests locaux.

09
Installation de phpMyAdmin

Sur le FRONT, installer phpMyAdmin pour administrer les bases de données :

apt install phpmyadmin # Ne rien cocher lors de l'installation, choisir NON pour dbconfig apt install php-fpm php-mbstring php-bcmath php-xml php-mysql \ php-common php-gd php-cli php-curl php-zip

Créer un virtual host Nginx pour phpMyAdmin sur le port 4443 avec SSL. Reconfigurer phpMyAdmin pour pointer vers BDD1 :

dpkg-reconfigure phpmyadmin # Choisir : Oui → TCP/IP → New host → IP de BDD1 → port 3306 → Default

Ajouter BDD2 dans la configuration pour pouvoir switcher entre les serveurs :

nano /etc/phpmyadmin/config.inc.php # Ajouter un second bloc $cfg['Servers'][$i] pointant vers 192.168.160.32

🔒 phpMyAdmin est accessible sur https://etu11.sio-carriat.com:4443/

10
Répartition de charge (Load Balancing)

Dupliquer WP1 en WP2 (clone VM), modifier le hostname et l'adresse IP :

hostnamectl set-hostname NLU-WP2-TPLABO nano /etc/hosts # Modifier la ligne pour NLU-WP2-TPLABO avec l'IP 192.168.160.22

Sur le FRONT, ajouter WP2 dans le bloc upstream pour activer le load balancing en round-robin (comportement par défaut) :

nano /etc/nginx/sites-available/wordpress upstream wordpress { server 192.168.160.21; server 192.168.160.22; }

Pour du load balancing par poids (weight), modifier le upstream :

upstream wordpress { server 192.168.160.21 weight=3; server 192.168.160.22 weight=1; }

Pour de la persistance de session (sticky sessions), utiliser ip_hash — garantit qu'un même client est toujours routé vers le même serveur :

upstream wordpress { ip_hash; server 192.168.160.21; server 192.168.160.22; }

Synchroniser les fichiers WordPress entre WP1 et WP2 avec Unison. Créer d'abord une paire de clés SSH :

# Sur WP1 : ssh-keygen -t ed25519 # Transmettre la clé publique à WP2 : cat ~/.ssh/wp1-wp2.pub | ssh root@192.168.160.22 \ "cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys" # Lancer la synchronisation manuellement : unison /var/www/html/wordpress/wp-content/uploads \ ssh://192.168.160.22//var/www/html/wordpress/wp-content/uploads

Automatiser la synchronisation toutes les 2 minutes via crontab :

crontab -e # Ajouter la ligne : */2 * * * * /usr/bin/unison default

⚠️ Autoriser SSH root sur WP2 pour que la clé fonctionne : nano /etc/ssh/sshd_config → ajouter PermitRootLogin yes

11
Réplication de la base de données (Maître-Maître)

Sur BDD1, créer l'utilisateur de réplication :

CREATE USER 'wp_replication'@'%' IDENTIFIED BY 'Azerty_123'; GRANT REPLICATION SLAVE ON *.* TO 'wp_replication'@'%'; FLUSH PRIVILEGES;

Configurer BDD1 comme maître dans le fichier de configuration MariaDB :

nano /etc/mysql/mariadb.conf.d/50-server.cnf # Ajouter : log_bin = /var/log/mysql/mysql-bin server_id = 1 binlog_do_db = NLU_Wordpress
mkdir /var/log/mysql/mysql-bin.index chown -R mysql:mysql /var/log/mysql chmod 750 /var/log/mysql systemctl restart mariadb.service # Vérifier le statut maître : mariadb SHOW MASTER STATUS;

Configurer l'esclave sur BDD1 (qui pointe vers BDD2) en récupérant le fichier log et la position depuis BDD2 :

STOP SLAVE; CHANGE MASTER TO MASTER_HOST = '192.168.160.32', MASTER_USER = 'wp_replication', MASTER_PASSWORD = 'Azerty_123', MASTER_LOG_FILE = 'mysql-bin.000001', MASTER_LOG_POS = 328; START SLAVE;

Répéter la même procédure sur BDD2 (server_id = 2), en pointant vers BDD1 :

CHANGE MASTER TO MASTER_HOST = '192.168.160.31', MASTER_USER = 'wp_replication', MASTER_PASSWORD = 'Azerty_123', MASTER_LOG_FILE = 'mysql-bin.000002', MASTER_LOG_POS = 328; START SLAVE;

♻️ Réplication maître-maître : BDD1 et BDD2 se répliquent mutuellement. Toute écriture sur l'un est propagée à l'autre automatiquement.

12
Haute disponibilité BDD avec Keepalived (VIP)

Sur BDD1 et BDD2, installer Keepalived pour gérer une IP virtuelle partagée :

apt install keepalived nano /etc/keepalived/keepalived.conf

Configuration sur BDD1 (MASTER) :

global_defs { enable_script_security } vrrp_script check_mariadb { script "/usr/bin/pgrep mariadb" interval 1 # vérifie toutes les secondes fall 2 # 2 échecs → DOWN rise 2 # 2 succès → UP } vrrp_instance VirtualIP { state MASTER interface ens33 virtual_router_id 1 priority 100 # 50 sur BDD2 authentication { auth_type PASS auth_pass MonMotDePasse } virtual_ipaddress { 192.168.160.30 # VIP partagée } track_script { check_mariadb } }
chmod 600 /etc/keepalived/keepalived.conf # Créer l'utilisateur système requis par Keepalived : useradd -U -M -s /sbin/nologin keepalived_script systemctl restart keepalived systemctl status keepalived

Sur BDD2, utiliser la même configuration avec state BACKUP et priority 50.

Mettre à jour les fichiers wp-config.php de WP1 et WP2 pour pointer vers la VIP :

define('DB_HOST', '192.168.160.30'); # Les deux serveurs web se connectent désormais à la VIP. # En cas de panne de BDD1, la VIP bascule automatiquement sur BDD2.

🟢 Fonctionnement : Si MariaDB s'arrête sur BDD1, le script pgrep mariadb échoue 2 fois → Keepalived baisse la priorité de BDD1 → BDD2 (priorité 50) devient MASTER et hérite de la VIP 192.168.160.30 → WordPress continue de fonctionner sans interruption.
