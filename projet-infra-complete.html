Configuration routeur & commutateur Cisco

Le switch Cisco est configuré avec 7 VLANs métier (71 à 77), un VLAN serveurs (407) et un VLAN internet (207). Le port G0/1 est configuré en TRUNK. Le routeur effectue du routage inter-VLAN via des sous-interfaces et du routage par origine (policy-based routing) pour diriger le trafic des VLANs vers pfSense.

# Access-list pour les réseaux internes access-list 10 permit 172.16.71.0 0.0.0.255 access-list 10 permit 192.168.7[1-7].0 0.0.0.255 # Route-map vers pfSense route-map LUONG07 permit match ip address 10 set ip next-hop 172.18.71.1 # Application sur toutes les sous-interfaces sauf G0/0.207 interface g0/0.[SOUS_INTERFACE] ip policy route-map LUONG07

2 — Connexion internet, pfSense & VPN

Configuration pfSense & routes statiques

pfSense est configuré avec deux interfaces : WAN (EM0 — 172.20.107.253/24) et LAN (EM1 — 172.18.71.1/30) connectée au routeur Cisco. Une route statique temporaire est ajoutée via le shell pfSense pour permettre l'accès initial au WebConfigurator depuis les VMs.

# Shell pfSense (option 8) — route temporaire route add -net 172.16.71.0/24 172.18.71.2

Des routes statiques persistantes sont ensuite créées via System > Routing > Static Routes pour joindre les VLANs internes. La règle LAN est modifiée pour autoriser Any en source. Le redirecteur DNS Windows est configuré vers 8.8.8.8. Une redirection NAT RDP (port 3389) permet d'accéder au serveur AD depuis l'extérieur via l'IP WAN du pfSense.

PKI, LDAPS & intégration AD pfSense

Depuis 2022, pfSense utilise LDAPS (port 636) pour se connecter à l'AD. Une PKI d'entreprise est déployée sur le contrôleur de domaine pour émettre les certificats nécessaires.

  • 01
    Déploiement de l'autorité de certification (AD-CA) — Installation du rôle AD-Certificate et ADCS-Cert-Authority, configuration en CA racine d'entreprise HPK07-ROOT-CA (RSA 2048, SHA256, validité 5 ans).
  • 02
    Activation LDAPS — Ajout du modèle DomainController via certutil -setcatemplates +DomainController, installation automatique du certificat via certutil -pulse.
  • 03
    Export du certificat racine — Export en .cer puis ré-encodage en .pem (base64) via certutil pour import dans pfSense.
  • 04
    Création des OUs et groupes AD — OU Sécurité avec groupe "Observateurs pfSense", utilisateur de liaison pfsense.connect.
  • 05
    Configuration pfSense — Import du certificat CA, ajout du serveur LDAPS dans System > User Manager, création du groupe "Observateurs pfSense" avec les droits dashboard, test via Diagnostics > Authentication.
# Installation PKI Install-WindowsFeature -Name AD-Certificate, ADCS-Cert-Authority Install-AdcsCertificationAuthority -CAType EnterpriseRootCA ` -CACommonName "HPK07-ROOT-CA" -KeyLength 2048 ` -HashAlgorithm SHA256 -ValidityPeriod Years -ValidityPeriodUnits 5 # Création OU et groupes AD New-ADOrganizationalUnit -Path "DC=hpk07,DC=labo" -Name "Sécurité" -PassThru New-ADGroup -Name 'Observateurs pfSense' -GroupCategory 'Security' ` -GroupScope 'DomainLocal' -Path 'OU=Sécurité,DC=hpk07,DC=labo' New-ADUser -Name "pfsense.connect" -Path "OU=Sécurité,DC=HPK07,DC=labo" ` -AccountPassword (ConvertTo-SecureString "Carriat_1234" -AsPlainText -Force) ` -Enabled $true -PasswordNeverExpires $true

VPN OpenVPN avec authentification Active Directory

Déploiement d'un serveur VPN OpenVPN sur pfSense avec authentification des utilisateurs via le groupe AD VPN User. Les utilisateurs VPN sont isolés dans leur propre OU AD.

  • 01
    OU et groupe VPN AD — Création de l'OU VPN et du groupe "VPN User" en PowerShell. Les utilisateurs autorisés sont ajoutés à ce groupe.
  • 02
    Certificats pfSense — Création d'une autorité de certification pfSense, d'un certificat serveur OpenVPN et d'un certificat utilisateur via System > Certificates.
  • 03
    Configuration serveur OpenVPN — Plage tunnel VPN dédiée, réseaux internes accessibles via VPN, connexions simultanées limitées, Dynamic IP activé (maintien connexion sur changement d'IP), option auth-nocache pour la sécurité, diffusion du DNS interne aux clients.
  • 04
    Export et déploiement client — Installation du package openvpn-clientexport sur pfSense, export du profil "Inline Configurations – Most Clients", déploiement dans C:\Program Files\OpenVPN Connect\config\.
  • 05
    Règles de pare-feu VPN — Création des règles Firewall > Rules permettant le trafic depuis le réseau tunnel vers les ressources internes.
# Création OU et groupe VPN New-ADOrganizationalUnit -Path "DC=hpk07,DC=labo" -Name "VPN" -PassThru New-ADGroup -Name 'VPN User' -GroupCategory 'Security' ` -GroupScope 'DomainLocal' -Path 'OU=VPN,DC=hpk07,DC=labo' ` -Description 'Utilisateur autorisé pour connexion VPN' -PassThru

3 — Stockage réseau — TrueNAS

Déploiement d'un serveur NAS TrueNAS intégré au domaine Active Directory. Configuration d'un pool de stockage en RAID1 (Mirror) avec trois datasets — SMB, FTP et TFTP — chacun avec des permissions adaptées aux groupes AD.

  • 01
    Configuration réseau — Attribution d'une IP statique (172.16.71.3), hostname, domaine, passerelle et DNS. Activation du redirect HTTP → HTTPS dans System > General Settings.
  • 02
    Jonction au domaine AD — Via Credentials > Directory Services > Configure Active Directory. Renseignement du domaine hpk07.labo, compte d'administration, mot de passe et NetBIOS name.
  • 03
    Création du pool RAID1 — Ajout de 2 disques supplémentaires, création du pool en mode Mirror (RAID1) via Storage > Create Pool.
  • 04
    Dataset SMB — Créé avec le preset SMB. Permissions NTFS : utilisateurs du domaine ont le contrôle total. Accessible via \\LUONG-HPK-NAS\SMB.
  • 05
    Dataset FTP — Permissions différenciées : utilisateurs du domaine en lecture/exécution uniquement, admins du domaine en contrôle total.

🔒 Intégration AD : La jonction TrueNAS au domaine hpk07.labo permet d'utiliser directement les comptes et groupes Active Directory pour gérer les permissions sur les datasets, sans créer de comptes locaux supplémentaires.

4 — ERP Odoo 19 & serveur web Nginx

Déploiement d'Odoo 19 Community sur un serveur Debian 13 dédié (172.16.71.11), connecté à la base de données PostgreSQL distante (172.16.71.4), exposé via un reverse proxy Nginx.

Installation Odoo 19

# Dépendances système apt install git python3-pip build-essential wget python3-dev \ python3-venv python3-wheel libxslt-dev libzip-dev libldap2-dev \ libsasl2-dev python3-setuptools node-less libjpeg-dev zlib1g-dev \ libpq-dev postgresql-client -y # Utilisateur et installation Odoo useradd -m -d /opt/odoo -U -r -s /bin/bash odoo su - odoo git clone https://github.com/odoo/odoo --depth 1 --branch 19.0 python3 -m venv venv && source venv/bin/activate pip install wheel && pip install -r odoo/requirements.txt

Configuration Odoo (/etc/odoo.conf)

[options] admin_passwd = carriat db_host = 172.16.71.4 db_port = 5432 db_user = odoo db_password = Carriat_1234 addons_path = /opt/odoo/odoo/addons logfile = /var/log/odoo/odoo.log proxy_mode = True

Service systemd

[Unit] Description=Odoo After=network.target [Service] User=odoo ExecStart=/opt/odoo/venv/bin/python3 /opt/odoo/odoo/odoo-bin -c /etc/odoo.conf [Install] WantedBy=multi-user.target
systemctl daemon-reload && systemctl enable odoo && systemctl start odoo

Reverse proxy Nginx

# /etc/nginx/sites-available/odoo server { listen 80; server_name _; proxy_read_timeout 720s; proxy_connect_timeout 720s; proxy_send_timeout 720s; client_max_body_size 100M; location / { proxy_pass http://127.0.0.1:8069; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } }
ln -s /etc/nginx/sites-available/odoo /etc/nginx/sites-enabled/ systemctl restart nginx

🌐 Accès Odoo : Odoo est accessible en interne sur http://172.16.71.11 via Nginx (port 80), qui proxifie vers le service Odoo sur le port 8069. PhpPgAdmin est accessible sur http://172.16.71.4/phppgadmin.
