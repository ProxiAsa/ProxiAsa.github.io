02 — Connexion internet, pfSense & VPN

Configuration du pare-feu pfSense

pfSense est configuré avec deux interfaces : EM0 (WAN) sur 172.20.107.253/24 et EM1 (LAN) sur 172.18.71.1/30, connectée au routeur Cisco. Pour permettre l'accès initial au WebConfigurator depuis les VMs internes, une route temporaire est ajoutée via le shell pfSense (option 8 du menu) :

Shell pfSense (option 8) — route temporaire
route add -net 172.16.71.0/24 172.18.71.2

Accès WebConfigurator : http://172.18.71.1 — login : Admin / mdp : pfsense

Des routes statiques persistantes sont ensuite créées via System > Routing > Gateways (ajout de la gateway vers le routeur Cisco), puis System > Routing > Static Routes pour joindre le réseau 172.16.71.0/24 et les VLANs 192.168.7x.0/24.

Accès internet & routage par origine

Pour activer l'accès internet depuis les VLANs, deux modifications sont nécessaires sur pfSense :

  • pfSense
    Interface WAN — décocher les cases de blocage des réseaux privés et bogons entrants (Interfaces > WAN).
  • pfSense
    Règle Firewall LAN — Firewall > Rules > LAN > IPv4 : mettre Any en source pour autoriser tout le trafic LAN vers internet.

Sur le routeur Cisco, le routage par origine (policy-based routing) est configuré pour rediriger le trafic des VLANs internes vers pfSense :

Routeur Cisco — Policy-Based Routing
! Access-list pour tous les réseaux internes access-list 10 permit 172.16.71.0 0.0.0.255 access-list 10 permit 192.168.7[1-7].0 0.0.0.255 ! Route-map : trafic matché envoyé vers pfSense route-map LUONG07 permit match ip address 10 set ip next-hop 172.18.71.1 ! Application sur toutes les sous-interfaces sauf G0/0.207 (internet) interface g0/0.71 ip policy route-map LUONG07 interface g0/0.72 ip policy route-map LUONG07 ! ... répéter pour .73 à .77 et .407

Redirecteur DNS & redirection RDP

Redirecteur DNS : Sur le contrôleur de domaine, ouvrir les outils DNS, clic droit sur le serveur > Propriétés > onglet Redirecteurs. Ajouter l'IP 8.8.8.8 (DNS Google) pour la résolution des noms externes.

Redirection RDP : Depuis l'interface pfSense, créer une règle NAT (Firewall > NAT > Add Forward) pour rediriger le port 3389 de l'interface WAN vers l'adresse IP du serveur AD. Cela permet l'accès RDP au serveur AD depuis l'extérieur via l'IP WAN du pfSense.

PKI, LDAPS & intégration AD dans pfSense

Depuis 2022, pfSense utilise exclusivement LDAPS (port 636, chiffré TLS) pour se connecter à l'Active Directory. Il est donc nécessaire de déployer une PKI interne sur le contrôleur de domaine pour émettre un certificat LDAPS valide.

Étape 1 — Déploiement de la PKI (AD-CA)

PowerShell — Installation des rôles PKI
Install-WindowsFeature -Name AD-Certificate, ADCS-Cert-Authority

AD-Certificate installe les composants de base de gestion des certificats dans Active Directory. ADCS-Cert-Authority installe le rôle d'Autorité de Certification (CA), composant central permettant d'émettre, gérer et révoquer des certificats numériques.

PowerShell — Configuration de la CA racine d'entreprise
Install-AdcsCertificationAuthority -CAType EnterpriseRootCA ` -CACommonName "HPK07-ROOT-CA" ` -KeyLength 2048 ` -HashAlgorithm SHA256 ` -ValidityPeriod Years ` -ValidityPeriodUnits 5

Détail des paramètres :

  • -CAType
    EnterpriseRootCA — CA racine d'entreprise, première CA de la hiérarchie PKI, intégrée à l'AD.
  • -CACommonName
    HPK07-ROOT-CA — Nom commun (CN) visible dans les certificats émis.
  • -KeyLength
    2048 bits — Bon équilibre sécurité/performance. La valeur 4096 est plus sécurisée mais impacte les performances.
  • -HashAlgorithm
    SHA256 — Algorithme de hachage recommandé pour signer les certificats.
  • -ValidityPeriodUnits
    5 ans — Durée de validité du certificat racine.

Étape 2 — Activation LDAPS & export du certificat

Activation du modèle DomainController pour LDAPS
# Ajoute le modèle DomainController à la liste des modèles émettables par la CA certutil -setcatemplates +DomainController # Installation automatique du certificat sur le contrôleur de domaine certutil -pulse
PowerShell — Export du certificat racine au format PEM
# Lister les certificats racine et trouver l'empreinte de HPK07-ROOT-CA Get-ChildItem -Path Cert:\LocalMachine\Root # Récupérer le certificat par son empreinte (remplacer THUMBPRINT) $cert = Get-ChildItem -Path Cert:\LocalMachine\Root\THUMBPRINT # Exporter en .cer vers un dossier partagé Export-Certificate -Cert $cert -FilePath \\LUONG-HPK-ADDS\Cert\root-ca-cert.cer -Type CERT # Ré-encoder en .pem (base64) pour import dans pfSense certutil -encode \\LUONG-HPK-ADDS\Cert\root-ca-cert.cer \\LUONG-HPK-ADDS\Cert\root-ca-cert.pem

Étape 3 — Préparation AD : OUs, groupes et utilisateur de liaison

PowerShell — Création OU Sécurité et groupe Observateurs pfSense
# Création de l'OU Sécurité New-ADOrganizationalUnit -Path "DC=hpk07,DC=labo" -Name "Sécurité" -PassThru # Création du groupe de sécurité pour l'accès au dashboard pfSense New-ADGroup -Name 'Observateurs pfSense' ` -GroupCategory 'Security' ` -GroupScope 'DomainLocal' ` -Path 'OU=Sécurité,DC=hpk07,DC=labo' ` -Description 'Groupe de surveillance pfSense' -PassThru # Création de l'utilisateur de liaison pfSense ↔ AD New-ADUser -Name "pfsense.connect" ` -SamAccountName "pfsense.connect" ` -UserPrincipalName "pfsense.connect@hpk.labo" ` -Path "OU=Sécurité,DC=HPK07,DC=labo" ` -AccountPassword (ConvertTo-SecureString "Carriat_1234" -AsPlainText -Force) ` -Enabled $true ` -PasswordNeverExpires $true ` -CannotChangePassword $true

Étape 4 — Configuration pfSense côté LDAPS

  • 01
    System > General Setup — Renseigner les informations du domaine, décocher DNS Server Override.
  • 02
    Services > DNS Resolver — Désactiver le service DNS Resolver pour éviter les conflits avec le DNS AD.
  • 03
    System > Certificates > Authorities — Importer le certificat racine CA en collant le contenu du fichier root-ca-cert.pem exporté depuis l'AD.
  • 04
    System > User Manager > Authentication Servers — Ajouter le serveur LDAP(S) : type LDAP, protocole LDAPS (port 636), serveur = IP de l'AD, CA = HPK07-ROOT-CA importé, décocher Bind Anonymous, renseigner pfsense.connect et son mot de passe.
  • 05
    Groupe pfSense — Créer un groupe dont le nom est exactement Observateurs pfSense (identique au groupe AD), lui attribuer les droits d'accès au dashboard.
  • 06
    Diagnostics > Authentication — Tester avec un compte membre du groupe AD. Le groupe Observateurs pfSense doit apparaître dans la fenêtre verte, confirmant l'association utilisateur ↔ groupe ↔ droits.
  • 07
    System > User Manager > Settings — Modifier la méthode de connexion pour utiliser le serveur AD. Save & Test.

✅ Résultat : Les utilisateurs membres du groupe Observateurs pfSense dans l'AD peuvent désormais se connecter au WebConfigurator pfSense avec leurs identifiants de domaine, avec les droits définis sur le groupe pfSense correspondant.

VPN OpenVPN avec authentification Active Directory

Déploiement d'un serveur VPN OpenVPN sur pfSense, avec authentification des utilisateurs via le groupe AD VPN User hébergé dans une OU dédiée.

Préparation AD — OU et groupe VPN

PowerShell — Création OU VPN et groupe VPN User
New-ADOrganizationalUnit -Path "DC=hpk07,DC=labo" -Name "VPN" -PassThru New-ADGroup -Name 'VPN User' ` -GroupCategory 'Security' ` -GroupScope 'DomainLocal' ` -Path 'OU=VPN,DC=hpk07,DC=labo' ` -Description 'Utilisateur autorisé pour connexion VPN' -PassThru

Les utilisateurs autorisés à se connecter au VPN sont créés dans l'OU VPN et ajoutés au groupe VPN User. La connexion LDAPS de pfSense est également mise à jour pour inclure l'OU VPN.

Création des certificats pfSense

  • CA
    System > Certificates > Authorities > Add — Créer une autorité de certification interne à pfSense pour signer les certificats OpenVPN.
  • Serveur
    System > Certificates > Add/Sign — Créer un certificat serveur signé par la CA pfSense, type "Server Certificate".
  • Client
    Créer un certificat utilisateur (user certificate) pour les clients VPN.

Configuration du serveur OpenVPN

  • 01
    VPN > OpenVPN > Add — Sélectionner le serveur d'authentification AD, la CA pfSense et le certificat serveur.
  • 02
    IPv4 Tunnel Network — Plage d'adresses dédiée aux clients VPN (ex : 10.0.8.0/24).
  • 03
    IPv4 Local Networks — Renseigner tous les réseaux internes à rendre accessibles depuis le VPN (172.16.71.0/24, 192.168.71.0/24 ... 77.0/24).
  • 04
    Concurrent Connections — Nombre maximum d'utilisateurs connectés simultanément.
  • 05
    Dynamic IP — Cocher pour maintenir la connexion VPN lors d'un changement d'IP (utile pour les utilisateurs en 4G).
  • 06
    Topology — Laisser sur Subnet pour permettre plus de connexions simultanées. Mettre net30 pour isoler les clients entre eux.
  • 07
    DNS — Renseigner le domaine et l'IP du DNS interne (172.16.71.1) pour diffuser la résolution DNS AD aux clients VPN.
  • 08
    Custom Options — Ajouter auth-nocache pour interdire aux clients OpenVPN de mémoriser les identifiants.

Export de la configuration client & règles firewall

  • Export
    System > Package Manager > Available Packages — Installer le package openvpn-clientexport. Puis VPN > OpenVPN > Client Export, télécharger Inline Configurations – Most Clients. Remettre l'option auth-nocache dans les paramètres additionnels avant l'export. Placer le fichier dans C:\Program Files\OpenVPN Connect\config\.
  • Firewall
    Firewall > Rules > Add — Créer les règles autorisant le trafic depuis le réseau tunnel VPN vers les ressources internes.
  • Client
    Ouvrir OpenVPN Connect, Upload File, sélectionner le fichier .ovpn, entrer les identifiants AD du groupe VPN User.

03 — Stockage réseau — TrueNAS

Configuration réseau & accès web

Au démarrage de TrueNAS, refuser le DHCP et configurer une IP statique. Renseigner ensuite le hostname, le domaine (hpk07.labo), la passerelle (172.16.71.254) et le DNS (172.16.71.1) dans la section Network Settings.

Interface web accessible sur : http://172.16.71.3

Activer le redirect HTTP → HTTPS : System > General Settings > GUI Settings → cocher Web Interface HTTP → HTTPS Redirect.

Création du pool RAID1 & jonction AD

Ajouter 2 disques supplémentaires à la VM TrueNAS. Créer ensuite un pool de stockage en RAID1 (nommé Mirror dans TrueNAS) :

  • 01
    Storage > Create Pool — Sélectionner les 2 disques, choisir le type Mirror (RAID1), nommer le pool. Cliquer sur Save And Go To Review puis Create Pool.
  • 02
    Credentials > Directory Services > Configure Active Directory — Renseigner le nom de domaine (hpk07.labo), un compte pouvant se connecter au domaine (ex : Administrateur), son mot de passe, et le NetBIOS name de la machine. Cocher Enable et valider.

✅ Résultat jonction AD : TrueNAS apparaît comme membre du domaine hpk07.labo et peut utiliser directement les comptes et groupes Active Directory pour les permissions des datasets, sans créer de comptes locaux.

Datasets SMB & FTP avec permissions AD

  • SMB
    Storage > Add Dataset — Créer le dataset SMB avec le preset SMB. Les permissions NTFS générées donnent le contrôle total aux utilisateurs du domaine AD. Accessible depuis les postes via \\LUONG-HPK-NAS\SMB.
  • FTP
    Storage > Add Dataset — Créer le dataset FTP. Configurer les permissions manuellement : utilisateurs du domaine en lecture et exécution uniquement, admins du domaine en contrôle total.
  • TFTP
    Créer le dataset TFTP de la même façon, avec les permissions adaptées au service TFTP.

Vérification : Odoo accessible sur http://172.16.71.11:8069

Reverse proxy Nginx

Nginx est déployé sur le même serveur qu'Odoo pour exposer le service sur le port 80 et gérer le proxy vers le port interne 8069. Le paramètre proxy_mode = True dans odoo.conf est obligatoire pour que les headers X-Forwarded-For soient correctement interprétés.

Installation Nginx
apt install nginx -y
/etc/nginx/sites-available/odoo
server { listen 80; server_name _; proxy_read_timeout 720s; proxy_connect_timeout 720s; proxy_send_timeout 720s; client_max_body_size 100M; location / { proxy_pass http://127.0.0.1:8069; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } }
Activation du site et redémarrage Nginx
ln -s /etc/nginx/sites-available/odoo /etc/nginx/sites-enabled/ systemctl restart nginx

🌐 Accès final : Odoo est accessible via http://172.16.71.11 (Nginx port 80 → Odoo port 8069). PhpPgAdmin sur http://172.16.71.4/phppgadmin.
